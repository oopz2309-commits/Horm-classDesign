# AGENTS.md

## 项目协作规则

- 所有沟通、说明、计划和总结均使用中文。
- 开发前先阅读项目结构、相关 `.md` 文档和现有代码，确认当前阶段后再动手。
- 修改代码前说明本次会改哪些文件，保持最小改动，不做无关重构。
- 保护 `.env`、`application.yml`、API Key、Token、数据库密码等敏感信息，不提交到 GitHub。
- 给出方案时先说明权衡，再给出建议。

## 接口开发流程

- 每次只开发一个明确接口或一个高度相关的小接口组，避免一次性改动过大。
- 开发顺序优先遵循文档规划：用户模块、景点模块、美食模块、收藏模块、评论模块。
- 开发接口前先核对：
  - 需求分析文档中的功能目标。
  - 后端接口设计文档中的请求路径、请求参数和返回格式。
  - 数据库表设计文档中的字段名、字段类型和测试数据。
  - 当前实体类、Mapper、Service、Controller 的已有实现。
- 接口返回格式统一使用 `Result<T>`，保持 `code`、`message`、`data` 结构一致。
- 数据库字段使用下划线命名，Java 字段使用驼峰命名，依赖 MyBatis 的下划线转驼峰配置映射。
- 不在接口代码中硬编码数据库密码、Token 或本机绝对路径。

## 单接口完成标准

每完成一个接口，必须先完成测试审核，确认无误后再进入下一个接口开发。

测试审核至少包括：

- 编译检查：运行 Maven 编译，确认无编译错误。
- 接口测试：启动后端服务，用浏览器、Postman、Apifox 或命令行请求接口。
- 返回值检查：确认 HTTP 状态码、`Result` 结构、`message` 和 `data` 与接口设计一致。
- 数据库检查：涉及增删改查时，确认数据库实际数据变化符合预期。
- 异常场景检查：至少验证关键失败场景，例如用户名重复、账号密码错误、ID 不存在、参数为空等。

只有测试审核通过后，才可以提交并上传 GitHub。

## GitHub 提交流程

- 每完成并测试通过一个接口，创建一次清晰提交。
- 提交前运行 `git status`，确认没有把 `.idea/`、`target/`、`application.yml`、`.env` 或敏感文件加入暂存区。
- 提交信息使用中文，简洁说明本次接口或功能，例如：`实现用户登录接口`。
- 提交后推送到 GitHub，再继续开发下一个接口。
- 如果推送失败，先说明失败原因，不进行强制推送，除非用户明确授权。

## 依赖和外部操作审批

- 如果需要下载依赖、安装工具、访问外部网络、修改 Maven 配置或执行可能影响环境的命令，必须先向用户说明原因并请求审批。
- 未经确认，不主动安装新依赖，不修改全局环境，不更改用户的数据库账号密码配置。
- 如果某个操作需要提升权限，先说明用途和影响，再等待用户审批。

## 验证说明

- 每次修改完成后，说明实际改动内容和验证方式。
- 如果某项测试无法执行，明确说明原因和剩余风险。
- 如果发现文档、数据库和代码不一致，先指出差异，再给出建议，不擅自扩大修改范围。
